# Multi-Level Cache Simulator Design (Approved)

Date: 2026-03-03
Architecture decision: Option 1 (Vite + React + TypeScript static SPA on Vercel)
Audience: CS students

## 1) Goals and non-goals

### Goals
- Deliver a clarity-first, educational simulator for byte-addressed multi-level cache behavior.
- Support 1 to 3 cache levels (L1-L3) with consistent per-level configuration and deterministic simulation.
- Show step-by-step execution details, including tag/index/offset decode and set-level tag comparison.
- Support policy scope approved for v1: replacement (LRU, FIFO) and write policy behavior (write-through, write-back).
- Accept workloads from built-in examples and manual text input.
- Provide a desktop-first, two-column UI with functional mobile fallback.

### Non-goals
- No comparison mode in v1 (single simulation only).
- No collaboration, persistence, accounts, or backend services.
- No production-grade performance tuning beyond responsive educational use.
- No advanced policy families beyond approved scope.

## 2) System architecture

The app is a static SPA built with Vite + React + TypeScript and deployed to Vercel static hosting.

- Runtime: browser-only execution; no server compute path.
- Build artifacts: static HTML/CSS/JS generated by Vite.
- State strategy: centralized immutable simulation state with pure reducers/selectors for deterministic replay.
- Simulation model: event-driven step pipeline where each user operation expands into an ordered sequence of events.
- UI model: container/presentational split; visual components consume derived view models from simulation state.

High-level modules:
- `domain/`: types and invariants for cache hierarchy, blocks, sets, memory, workload ops, and policies.
- `engine/`: address decode, lookup/replace logic, write policy handling, cascade behavior, and step emission.
- `parser/`: workload text parsing and normalization.
- `state/`: reducer/actions/history cursor for step mode.
- `ui/`: controls, tables, event timeline, and stats panels.

## 3) Domain/state model

Core entities:
- Memory: byte-addressed logical memory backing store.
- CacheLevel: `{ id, enabled, totalSizeBytes, blockSizeBytes, associativity, replacementPolicy, writePolicy }`.
- Set: fixed-size collection of ways per level.
- Block/Line: `{ valid, dirty, tag, dataBytes, lruCounter/fifoOrder, lastTouchedStep }`.
- Stats: per-level hits/misses/evictions and global memory read/write counters.
- WorkloadOp:
  - Read: `{ kind: "R", address }`
  - Write: `{ kind: "W", address, value }`

Derived geometry per level:
- `numSets = totalSizeBytes / (blockSizeBytes * associativity)`
- `offsetBits = log2(blockSizeBytes)`
- `indexBits = log2(numSets)`
- `tagBits = addressWidth - offsetBits - indexBits`

State containers:
- `configState`: user-configured hierarchy and policy choices.
- `simState`: mutable simulation snapshot represented immutably each step.
- `timelineState`: ordered event log for current operation and run history cursor.
- `inputState`: raw workload text, parse diagnostics, and selected built-in example.

## 4) Simulation step pipeline (tag comparison and write-back cascades)

Each operation executes as a deterministic pipeline with emitted events for instruction and debugging.

1. Parse operation (already validated by parser).
2. Decode address into `tag/index/offset` for each active level.
3. Select set by decoded `index` in current level.
4. Compare requested `tag` against all valid ways in that set.
5. Record hit/miss result and event details.
6. Apply operation behavior:
   - On read hit: return data, update replacement metadata.
   - On read miss: continue to next level; memory on last-level miss.
   - On write hit:
     - Write-through: update cache line and propagate write downstream immediately.
     - Write-back: update cache line and mark dirty.
   - On write miss: follow selected write policy handling path.
7. If fill required, choose victim by replacement policy (LRU or FIFO).
8. If victim is dirty under write-back, generate downstream write-back event and cascade until absorbed by lower level or memory.
9. Install/refresh block in current level and update metadata.
10. Finalize stats and append summarized operation outcome.

Event log details per level access include:
- Level id, operation kind, decoded `tag/index/offset`.
- Compared ways list with `{ way, valid, tag, match }`.
- Access outcome (`hit` or `miss`) and chosen victim when applicable.
- Dirty eviction/write-back actions and downstream target.

## 5) UI composition/layout

Visual direction (approved): Clean Academic, slate + blue light theme, minimal functional dark mode, sans UI/body with monospace accents for addresses/tags/values, subtle instructional motion only.

Desktop-first two-column layout:
- Left column (controls and input):
  - Hierarchy configuration card(s):
    - Enabled levels (L1-L3)
    - Total size (bytes)
    - Block size (bytes)
    - Associativity
    - Replacement policy (LRU/FIFO)
    - Write policy (write-through/write-back)
  - Workload input card:
    - Built-in example selector
    - Manual workload textarea
    - Parse/Load action
  - Simulation controls:
    - Step, Run, Pause, Reset
    - Speed control (for run mode)
    - Theme toggle (light default, minimal dark)
- Right column (results and teaching surface):
  - Current operation and decoded fields panel
  - Event timeline panel (per-step detail)
  - Cache level visualization panels (set/way views with active highlights)
  - Statistics panel (hits, misses, evictions, memory traffic)

Mobile behavior:
- Functional fallback with vertical stacking, no redesign goal for v1.

## 6) Workload input format and parser behavior

Accepted workload format (explicit):
- One operation per line.
- Read: `R <address>`
- Write: `W <address> <value>`
- `<address>` and `<value>` may be hex (e.g., `0x1A`) or decimal (e.g., `26`).
- `#` comments and blank lines are ignored.

Parser behavior:
- Trim whitespace and normalize internal spacing.
- Skip empty/comment lines.
- Parse op code case-insensitively (`R`/`W` accepted).
- Validate token count and numeric fields; produce line-specific diagnostics.
- Reject partially valid writes (missing value) and malformed numerics.
- Return normalized workload list plus diagnostics; simulation cannot start while hard errors exist.

## 7) Validation and error-handling strategy

Configuration validation:
- Require positive power-of-two `blockSizeBytes` and set counts.
- Enforce geometry consistency from size/block/associativity relations.
- Enforce monotonic hierarchy constraints (higher level must be larger than lower level).

Input/runtime validation:
- Validate addresses/values for range and numeric format.
- Stop only the current operation on runtime invariant violation; keep app responsive.
- Emit user-facing, plain-language errors with exact line or stage context.

Error surfaces:
- Inline form errors for configuration fields.
- Workload parser diagnostics list with line numbers.
- Non-fatal engine alerts in event timeline.

## 8) Testing and verification strategy

Unit tests:
- Address decode (`tag/index/offset`) across representative geometries.
- Replacement policy behavior (LRU and FIFO victim selection).
- Write-through and write-back paths, including dirty eviction cascades.
- Parser acceptance/rejection for all documented workload forms.

Integration tests:
- Multi-step read/write traces across 1-3 levels with deterministic expected timeline snapshots.
- Stats consistency checks (hits/misses/evictions/memory traffic).

UI tests:
- Critical flow: configure, load workload, step through, inspect timeline/stats.
- Desktop layout and functional mobile fallback sanity checks.

Manual verification checklist:
- Confirm tag-compare event details are visible and correct.
- Confirm write-back cascade events appear in proper order.
- Confirm built-in examples and manual input paths both execute.

## 9) Deployment strategy for Vercel static hosting

- Host as static SPA via Vercel.
- Build command: `npm run build` (or equivalent package script).
- Output directory: Vite `dist/`.
- No serverless functions required.
- Configure SPA fallback routing to `index.html` if non-root routes are introduced.
- Keep all logic client-side to preserve Option 1 architecture.

## 10) Open decisions deferred for later

- Optional address-decode visual strip (binary bit partition bar) remains deferred.
- Any side-by-side comparison mode remains deferred beyond v1.
- Any advanced policy extensions beyond LRU/FIFO and write-through/write-back remain deferred.
